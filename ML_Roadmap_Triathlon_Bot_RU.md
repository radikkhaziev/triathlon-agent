Оригинальные 4 направления дополнены 2 новыми на основе анализа расширенного бандла статей из Academia.edu (Vladimir Medved et al., IoT/wearable research, AI sport training).

## Направление 1: Предсказание готовности

> Прогноз ежедневного readiness score (0–100) и тренировочной рекомендации (zone2_ok / zone1_long / zone1_short / skip) по физиологическим и субъективным сигналам. **Самое ценное и самое готовое к данным направление.**

### Академическая база

- **Eskofier et al.** (Embedded Classification of Perceived Fatigue): SVM и LDA достигают 88.3% точности классификации усталости всего по 3 фичам HRV + биомеханика.
- **Miab et al.** (MHfit — Mobile Health for Predicting Fitness): сигналы мобильного здоровья (ЧСС, сон, активность) предсказывают готовность атлета; контекст окружения добавляет ~10% точности.
- **Passfield et al.** (Transforming the performance of runners with AI): ансамбль RNN + LSTM + регрессия для построения «цифрового двойника» атлета. 9-компонентный Readiness Questionnaire (не только HRV) повышает точность предсказания готовности. Симуляция 5000 вариантов программ через Monte Carlo для выбора оптимальной нагрузки.
- **Ключевой вывод:** фичи вариабельности HRV — самый сильный предиктор усталости. Минимального набора фич (3–5) достаточно для высокой точности. Многомерная субъективная оценка (9 вопросов) значительно усиливает объективные данные.

### Доступные фичи из MCP

| Фича | Источник MCP | Роль в ML |
|---|---|---|
| HRV (RMSSD), среднее 7д/60д, CV, slope, SWC | `get_hrv_analysis` | Главный предиктор усталости |
| RHR, среднее 7д/30д, направление тренда, slope | `get_rhr_analysis` | Маркер перетренированности / болезни |
| Оценка сна, длительность, качество (1–5) | `get_wellness_range` | Индикатор способности к восстановлению |
| CTL, ATL, TSB, ramp_rate | `get_training_load` | Кумулятивная усталость / форма |
| Энергия настроения, тревожность (шкала 1–5) | `get_mood_checkins_tool` | Субъективный слой готовности |
| Banister recovery %, ESS | `get_wellness_range` | Модельный baseline восстановления |

### Рекомендуемый подход

**Модель:** Gradient Boosted Trees (XGBoost / LightGBM) для табличных ежедневных данных как базовый вариант. Для продвинутой версии — ансамбль RNN + LSTM + регрессия по подходу Digital Twin (Passfield et al.), где модель обучается на индивидуальных данных атлета и способна симулировать отклик на различные нагрузки.

**Целевая переменная:** `recovery_score` (регрессия) + `recovery_category` (классификация). Начинаем с существующего Banister-score как ground truth, потом уточняем по обратной связи атлета.

**Валидация:** Time-series кросс-валидация (walk-forward). Никогда не допускать утечку будущих данных в обучение.

### Фазы реализации

1. Накопить 60–90 дней выровненных ежедневных данных (HRV + RHR + сон + TSB + mood). Сейчас ~34 дня — нужно ещё 2–4 недели.
2. Feature engineering: скользящий z-score HRV, производная TSB, взаимодействие качество сна × длительность, лаговые фичи настроения (вчерашний mood → сегодняшняя готовность).
3. Обучить XGBoost-регрессор с LOOCV или walk-forward CV. Baseline: предсказание среднего `recovery_score` (бенчмарк MAE).
4. Деплой как MCP-инструмент: `get_ml_readiness_prediction` — возвращает score + топ-3 влияющих фичи (SHAP values).

## Направление 2: Анализ паттернов

> Обнаружение скрытых связей между тренировочным поведением, сигналами восстановления и результатами. **Это исследовательский ML — цель не предсказание, а генерация инсайтов.**

### Академическая база

- **Emig & Peltonen** (Human Running Performance from Real-World Big Data): дистанция тренировок коррелирует с результатом при r=0.82. Индекс выносливости (Ei) более предиктивен, чем VO2max. Оптимальный диапазон TRIMP: 2500–3000 для подготовки к марафону.
- **Havas et al.** (Athlete Training Decision Support): иерархическая кластеризация группирует схожие профили атлетов; ANN обнаруживает нелинейные связи тренировка-результат. VO2max factor используется как ключевой параметр для генерации индивидуальной программы.
- **Medved et al.** (Surface EMG based muscle fatigue evaluation): wavelet transform и RQA (Recurrence Quantification Analysis) для выявления нелинейных паттернов утомления. Применимо не только к EMG, но и к временным рядам HRV — может быть точнее стандартного RMSSD.
- **IoT System for Monitoring** (Monitoring and Analysing Physiological Variables): 4-уровневая IoT-архитектура (capture → NoSQL storage → analysis → visualization) с unsupervised кластеризацией для автоматического выявления паттернов перетренированности.
- **Ключевой вывод:** тренировочный импульс (объём × интенсивность) важнее сырого объёма. Индивидуальные паттерны отклика существенно различаются — персонализация обязательна. Нелинейные методы анализа (wavelet, RQA) могут выявить паттерны усталости, невидимые для линейных корреляций.

### Что искать

- **Лаг HRV-нагрузка:** через сколько дней после тяжёлой сессии падает HRV? Стабилен ли лаг? Статьи указывают на задержку 1–3 дня.
- **Цепочка сон-настроение-результат:** предсказывает ли плохой сон низкое настроение, а низкое настроение — плохое качество тренировки?
- **Видоспецифичное восстановление:** утомляет ли плавание иначе, чем бег? Per-sport CTL позволяет проверить.
- **Сигналы перетренированности:** одновременный рост RHR + падение HRV + падение mood = составной индикатор усталости. Какой порог предсказывает рекомендацию «skip»?
- **Оптимальное распределение тренировок:** какой CTL-микс (плавание/вело/бег) коррелирует с лучшей траекторией TSB?

### Рекомендуемый подход

**Кросс-корреляция с лагом:** скольжение тренировочной нагрузки относительно HRV/RHR со сдвигами 0–7 дней для нахождения индивидуальной задержки отклика.

**Кластеризация (K-Means / DBSCAN):** группировка тренировочных дней по профилю фич (нагрузка, восстановление, mood) для выявления «архетипов дней» — например, высокая-нагрузка-хорошее-восстановление vs. низкая-нагрузка-плохое-восстановление.

**Причинность по Грейнджеру / VAR-модели:** тестирование, вызывает ли тренировочная нагрузка изменения HRV (или наоборот) в индивидуальном временном ряду.

**Ассоциативные правила:** майнинг правил вида `IF TSB < -15 AND sleep_quality < 3 THEN mood_energy < 2 (80% confidence)`.

**Wavelet-анализ HRV:** применение непрерывного wavelet transform (CWT) к временным рядам HRV для выявления частотно-временных паттернов усталости. В отличие от линейных корреляций, wavelet захватывает нестационарные изменения — например, постепенное угасание вариабельности перед перетренированностью.

**RQA (Recurrence Quantification Analysis):** нелинейный метод анализа HRV, изначально разработанный для EMG (Medved et al.). Метрики: determinism, recurrence rate, entropy — индикаторы перехода от здорового хаоса к патологической регулярности в сердечном ритме.

### Фазы реализации

1. Экспорт 60+ дней выровненных ежедневных данных в pandas DataFrame (индекс по дате, все MCP-источники объединены).
2. Матрица корреляций + анализ лагов: Pearson r между всеми парами фич при лагах 0–7д. Визуализация heatmap.
3. Кластеризация тренировочных дней (K-Means, k=3–5). Интерпретация кластеров как тренировочных архетипов. Маппинг на исходы восстановления.
4. Построить дашборд «личные паттерны»: ключевые находки показываются как natural-language инсайты в боте.

## Направление 3: Адаптивная персонализация планов

> Динамическая корректировка тренировочных планов на основе состояния атлета, истории выполнения и прогнозируемой готовности. **Объединяет предсказание готовности с назначением тренировок.**

### Академическая база

- **Janssen et al.** (Inspirun E-Coach): Personal Running Profile (PRP) на основе RPE, темпа и ЧСС позволяет итеративно адаптировать план. Бегуны с персонализированными планами показывают более высокую вовлечённость и compliance.
- **López-Matencio et al.** (Ambient Intelligence for Running): k-NN классификация по средовым + физиологическим фичам достигает ~100% точности выбора подходящих тренировочных маршрутов/интенсивности.
- **Feely et al.** (Modelling Recreational Marathon Runners): Case-Based Reasoning (CBR) сопоставляет текущее состояние атлета с похожими историческими случаями. Multi-week фичи лучше single-week. Покрытие 60–80% при оптимальной дельте сопоставления.
- **Fister et al.** (On deploying the Artificial Sport Trainer): AST-Monitor сравнивает плановую и фактическую интенсивность в реальном времени (целевой vs фактический HR). Compliance tracking — автоматическая оценка, насколько атлет следовал плану — напрямую кормит reward-функцию для RL.
- **Ключевой вывод:** CBR идеален для коучинга, потому что интерпретируем — «Я рекомендую это, потому что атлеты как ты на этом этапе делали X». Compliance tracking (план vs. факт) — обязательный компонент адаптивной системы.

### Рекомендуемый подход

**Reinforcement Learning (контекстный бандит):** Action = предложенный тип/интенсивность тренировки. Context = readiness score + TSB + день недели + недавний compliance. Reward = качество восстановления на следующий день + обратная связь по настроению.

**Почему RL, а не supervised:** у нас нет размеченных данных «оптимальная тренировка». Но мы можем наблюдать результат каждой рекомендации и учиться, какие действия ведут к лучшему восстановлению и настроению в контексте этого атлета.

**Compliance Tracking (AST-Monitor):** после каждой тренировки автоматически сравнивать план с фактом: целевой HR vs. фактический, плановый TSS vs. выполненный, целевая длительность vs. реальная. Метрика compliance (0–100%) становится ключевым сигналом для reward-функции RL и для корректировки следующих тренировок.

**Monte Carlo оптимизация (Digital Twin):** симуляция 5000 вариантов тренировочных программ на 1–4 недели вперёд через обученную модель Digital Twin. Выбор программы с лучшим ожидаемым исходом (максимум CTL-прироста при минимуме риска перетренированности).

**Fallback (CBR):** пока RL-данных мало, используем case-based reasoning: храним кортежи (состояние, действие, результат) и извлекаем k ближайших прошлых состояний для выбора рекомендации.

### Фазы реализации

1. Определить пространство состояний: `[readiness_score, TSB, ATL, sleep_quality, mood_energy, day_of_week, days_since_rest]`.
2. Определить пространство действий: `[skip, zone1_easy, zone2_long, zone2_tempo, intervals, race_pace, strength]`. Замапить существующие тренировки на эти категории.
3. Определить reward: взвешенная сумма `recovery_score` следующего дня (+), `mood_energy` (+) и выполненная `training_load` vs. плановая (compliance).
4. Начать с epsilon-greedy контекстного бандита (просто, без deep RL). Логировать каждую тройку (state, action, reward). После 100+ точек модель начинает превосходить rule-based рекомендации.

## Направление 4: Предсказание результата гонки

> Прогноз финишного времени на целевых гонках по истории тренировок, траектории фитнеса и физиологическим маркерам. **Наиболее изученная ML-задача в академической литературе.**

### Академическая база

- **Feely et al.** (Explainable Race-Time Predictions): CBR + отбор фич достигает RMSE 14–16 минут (марафон) за 4 недели до старта. Ключевое значение имеют multi-week агрегированные фичи. Prefactual-объяснения («пробеги 10 км быстрее на следующей неделе — выиграешь 3%») повышают вовлечённость.
- **Emig & Peltonen** (Real-World Big Data): на 1.5M тренировочных активностей 2840 марафонцев линейные модели дают RMSE 2.0%. Индекс выносливости Ei и общая дистанция — топ-предикторы.
- **Research on intelligent analysis of running** (AI-based running analysis): сравнение RNN vs. LSTM vs. GRU на данных смарт-часов для предсказания финишного времени. **GRU показал лучшую точность** при меньшей вычислительной сложности. Биомеханический анализ: скорость = каденс × длина шага — полезно для оптимизации пейс-стратегии.
- **Ключевые фичи из литературы:** недельная дистанция, самая длинная пробежка, самый быстрый недавний темп на 10 км, траектория CTL, индекс выносливости, стабильность тренировок (дисперсия недельного объёма), каденс и длина шага (биомеханика).

### Что нужно

Это направление требует больше всего данных — в идеале результаты гонок + 12–16 недель структурированных тренировок перед каждой гонкой. Сейчас в MCP ~34 дня данных без результатов гонок. Это самый долгосрочный проект.

### Рекомендуемый подход

**Фаза 1 (сейчас):** построить data pipeline. Для каждой тренировочной недели считать: общую дистанцию, общий TSS, длительность самой длинной сессии, средний ЧСС, CTL, количество сессий по видам спорта.

**Фаза 2 (после 1-й гонки):** даже с одним результатом + 12 неделями истории можно обучить простую линейную регрессию: `predicted_time = f(CTL_at_race, total_distance_12w, longest_run, avg_weekly_TSS)`.

**Фаза 2.5 (GRU-модель):** обучить GRU-сеть на временных рядах тренировок (HR, темп, дистанция, каденс) для предсказания финишного времени. GRU показывает лучшую точность при меньшей сложности, чем LSTM (Research on intelligent analysis of running). Входные данные: последовательность недельных агрегатов за 12 недель.

**Фаза 3 (после 3+ гонок):** добавить CBR-слой: сравнивать текущий тренировочный блок с собственными прошлыми предгоночными блоками и предсказывать по ближайшему совпадению.

### Фазы реализации

1. Создать агрегатор недельных тренировочных сводок: дистанция, TSS, количество сессий, самая длинная сессия, распределение по видам из `get_activities`.
2. Добавить логирование результатов гонок в MCP (целевое время, фактическое время, условия, RPE).
3. После первой гонки: обучить baseline линейную модель. После 3+ гонок: добавить CBR с prefactual-объяснениями.
4. Интегрировать в бота: «При текущей траектории (CTL 45, дистанция за 12 нед. 520 км), прогноз на полумарафон: 1:52 ± 5 мин.»

## Направление 5: Цифровой двойник атлета (Digital Twin)

> Индивидуальная ML-модель, симулирующая физиологический отклик конкретного атлета на тренировочные стимулы. **Самое амбициозное направление — объединяет предсказание готовности, персонализацию планов и прогноз результата в единую модель.**

### Академическая база

- **Passfield et al.** (Transforming the performance of runners with AI): ансамбль RNN + LSTM + регрессия обучается на индивидуальных данных одного атлета. Модель принимает на вход тренировочный план и симулирует, как изменятся CTL, ATL, HRV, готовность. Позволяет прогонять 5000 вариантов программ и выбирать оптимальную.
- **Havas et al.** (Modelling of the Athlete's Training Decision Support): VO2max factor как ключевой параметр модели. Neural networks предсказывают способность атлета достичь целевого результата. Data Warehouse архитектура для накопления исторических данных.

### Архитектура Digital Twin

**Входы модели:** текущее состояние атлета (HRV, RHR, CTL, ATL, TSB, сон, mood) + предлагаемая тренировка (тип, длительность, интенсивность, вид спорта).

**Выходы модели:** прогноз состояния на следующий день (HRV, recovery_score, mood_energy, CTL, ATL, TSB).

**Ключевое преимущество:** модель можно «раскрутить» вперёд на 1–4 недели, подавая на вход цепочку тренировок, и получить прогноз траектории фитнеса. Это превращает планирование из rule-based в simulation-based.

### Рекомендуемый подход

**Модель:** Multi-output регрессия на основе LSTM. Вход — скользящее окно 7–14 дней ежедневных фич + вектор тренировки. Выход — вектор состояния на следующий день. При малом количестве данных (< 90 дней) начать с линейной авторегрессионной модели (VAR), затем перейти к LSTM при 6+ месяцах данных.

**Симуляция программ:** для каждого дня горизонта генерировать N случайных тренировок из пространства действий, прогонять через модель, оценивать результат по целевой функции (например: максимум CTL на день гонки при TSB > -10).

**VO2max estimation:** косвенная оценка VO2max через пороговые данные Intervals.icu (HRVT2, пороги мощности). Использовать как нормализующий фактор для сравнения нагрузок разных атлетов и для калибровки модели.

### Фазы реализации

1. Построить авторегрессионную VAR-модель на существующих данных (60+ дней). Предсказать HRV/recovery на +1 день по скользящему окну. Это baseline для Digital Twin.
2. Добавить вектор тренировки как экзогенную переменную (VARX). Проверить, улучшается ли предсказание.
3. При накоплении 6+ месяцев данных перейти к LSTM. Multi-output: предсказать одновременно HRV, recovery, mood на +1 день.
4. Реализовать симулятор программ: Monte Carlo генерация 1000+ вариантов тренировочных недель, ранжирование по целевой функции, вывод топ-3 рекомендаций с объяснениями.

## Направление 6: Compliance и биомеханический анализ

> Автоматическая оценка соответствия тренировки плану и анализ биомеханических параметров для оптимизации техники. **Связывает качество выполнения тренировок с результатом.**

### Академическая база

- **Fister et al.** (On deploying the Artificial Sport Trainer): AST-Monitor — система реального времени для сравнения плановой и фактической интенсивности. Использует TSS/TRIMP-индикаторы нагрузки. GUI адаптирован под велосипедистов — применимо к триатлету.
- **Research on intelligent analysis of running** (AI running analysis): биомеханический анализ бега через каденс и длину шага. Формула: скорость = каденс × длина шага. GRU-модель предсказывает технические параметры по текущим данным.
- **A Sensor Platform for Athletes' Training** (Sensor Platform): PLS (Partial Least Squares) и PCA для построения моделей физиологических параметров по косвенным данным. Применимо для оценки лактатного порога без инвазивных тестов.
- **A Telemonitoring Solution to Long-Distance Running** (Telemonitoring): GPS + HR данные для пост-тренировочной аналитики. Расширенные Post-Run Summary экраны.

### Компоненты

**Compliance Score (план vs. факт):** после каждой тренировки бот автоматически сравнивает запланированные параметры с фактическими из `get_activity_details`: целевой HR-диапазон vs. фактический средний HR, плановый TSS vs. фактический, целевая длительность vs. реальная. Compliance score (0–100%) вычисляется как взвешенная метрика по всем параметрам.

**Биомеханический мониторинг бега:** отслеживание каденса и вертикальных осцилляций (если доступны через Garmin). Выявление деградации техники при утомлении: падение каденса + рост вертикальных осцилляций = сигнал усталости. Рекомендации по оптимизации каденса для длинных дистанций.

**Пейс-стратегия:** анализ распределения темпа по сегментам (negative/positive/even split). Выявление индивидуального оптимального паттерна на основе исторических данных. GRU-модель для предсказания оптимального пейса по конкретному маршруту (если доступен профиль высот).

### Рекомендуемый подход

**Compliance Score:** rule-based расчёт на первом этапе. Формула: `compliance = 0.4 × hr_match + 0.3 × tss_match + 0.2 × duration_match + 0.1 × type_match`, где каждый компонент = 1 - |план - факт| / план. Позже — обучить модель весов по обратной связи атлета.

**Биомеханика:** кросс-корреляция каденса с HR и темпом для выявления «зоны оптимального каденса». Сравнение каденса первой и последней трети тренировки как индикатор утомления.

### Фазы реализации

1. Реализовать compliance score как MCP-инструмент: `get_workout_compliance(activity_id, planned_workout_id)`. Сравнение запланированной и выполненной тренировки.
2. Добавить compliance в пост-тренировочный feedback бота. Использовать compliance history для корректировки будущих рекомендаций (если атлет систематически перевыполняет — скорректировать планку вверх).
3. Реализовать биомеханический мониторинг: каденс-трекинг для беговых активностей, выявление деградации техники при утомлении.
4. Интегрировать compliance score в reward-функцию RL (Направление 3) и в данные для Digital Twin (Направление 5).

# Workout Absolute Targets Spec

> Convert `%lthr` / `%ftp` / `%pace` workout step targets to absolute units
> (bpm / watts / sec-per-distance) at push-to-Intervals boundary, so watch
> alerts are independent of local Garmin/Coros/Wahoo zone configuration.
>
> Status: 🟢 **SHIPPED 2026-05-12 — root cause was field-name rename, not absolute conversion.**
> Intervals.icu workout_doc HR corridor schema uses field name **`start`**, not
> `value`. Our codebase historically emitted `{value: X, end: Y}` — Intervals
> API accepted it (lenient validation) but FIT export interpreted such steps as
> «Lap HR / zone-mapped» which watch then clamped to its own zone boundaries.
> Steps with `{start: X, end: Y}` get FIT-exported as proper absolute corridors,
> watch displays exact bpm ranges. Empirically verified §12 «Attempt 3b — success».
>
> **Single-character fix** at code level: rename `value` → `start` in HR/power/
> pace dicts emitted by `bot/prompts.py` + accepted by `data/intervals/dto.py`
> validator/renderer. No server-side `%X → absolute` computation needed; Intervals
> handles translation correctly once schema is right. Phase A («server-side
> converter») is **explicitly не делаем** — see §13.5 decision matrix.
>
> Two open Intervals UI quirks documented §13.6 (Swim min/km display hardcode +
> `update_event` no-enrichment) — not blockers, not our bugs.
>
> Triggered by issue: user discovered Garmin local LTHR=168 (auto-detect drift)
> vs system HRVT2=172, all zones silently off by ~2.3%. Root cause turned out to
> be wrong schema field name in our emitted payload, not watch-side drift.

---

## 1. Problem

Currently per-sport convention (CLAUDE.md, `bot/prompts.py:Workout generation`):

| Sport | Target type | Units |
|---|---|---|
| Run | `hr` | `%lthr` |
| Ride | `power` | `%ftp` |
| Swim | `pace` | `%pace` |

We push these to Intervals.icu (`actor_push_workout` / `IntervalsClient.create_event`),
Intervals exports to Garmin via FIT file (or Garmin Connect Calendar sync). The
watch translates `%lthr × LTHR_local = bpm_displayed` using **its own copy of
LTHR**.

**Failure mode (verified 2026-05-12 on user 1):**
- System LTHR=172 (synced from HRVT2 via `actor_update_zones` per #313)
- Watch LTHR=168 (Garmin auto-detect re-computed from recent activities, ignoring user's manual entry of 172)
- Workout step «Z2 main 85-89% LTHR» → expected `172 × 0.85-0.89 = 146-153 bpm`
- Watch displays: **143-149 bpm** (using 168×%)
- All zones silently shifted down ~2.3% → threshold sessions run at tempo intensity, base sessions at recovery intensity

**Root cause:** watch's local LTHR config drifts via:

1. **Auto-detect LTHR** — Garmin's «Detect Heart Rate Zones» feature recomputes LTHR from recent activities, can overwrite user's manual entry without notification.
2. **Per-profile config** — Garmin allows per-sport LTHR overrides (Running profile vs Cycling profile). User updates global, sport profile stays stale.
3. **Manual entry stale** — user updates LTHR in Garmin Connect, watch sync delay or cache issue keeps old value.
4. **Cross-vendor variation** — Coros/Wahoo/Polar each compute zones from LTHR differently (some use HR-reserve / Karvonen, some use %LTHR with their own zone boundaries).

System has no visibility into watch's actual LTHR. Cannot detect drift. Atлет silently trains in wrong zones.

## 2. Current state (post-fix)

```python
# data/intervals/dto.py — WorkoutStepDTO
{
    "text": "Z2 main",
    "duration": 1500,
    "hr": {"units": "%lthr", "start": 85, "end": 89},
}
```

Pre-fix the corridor used field name `value`; see §12 for the empirical
investigation that switched it to `start`.

- Generated by Claude (chat path) or `compose_workout` MCP tool, validated by
  `PlannedWorkoutDTO._check_steps_have_targets` (rejects steps without
  `hr/power/pace` for non-`Other` sports).
- Pushed to Intervals via `IntervalsClient.create_event` body → `workout_doc.steps[]`.
- Intervals stores `%lthr/%ftp/%pace` literally; on Garmin sync, translates via
  athlete's Intervals-side threshold values into absolute FIT-file targets.
- Athlete's Garmin watch receives FIT, applies its **own** local zone config
  to display / alert — **point where drift bites**.

## 3. Proposed solution

### 3.1. Approach

Add **server-side converter** at the push-to-Intervals boundary. Claude
continues to emit `%lthr/%ftp/%pace` (semantic, zone-readable). We resolve
to absolute units using `athlete_settings` thresholds before serialization.

```python
# data/intervals/dto.py — new method on WorkoutStepDTO
def to_absolute_target(self, *, lthr: int | None, ftp: int | None,
                       css_sec_per_100m: int | None,
                       run_threshold_pace_sec_per_km: int | None) -> dict:
    """Resolve percentage-based target to absolute units.

    Returns target dict ready for Intervals.icu workout_doc serialization.
    If threshold value is None (cold-start athlete) — return original %X
    target unchanged (fallback to current behavior).
    """
```

Call site: `actor_push_workout` (or wherever WorkoutStepDTO → Intervals
payload happens). For each step in `workout_doc.steps`, replace `hr/power/pace`
with absolute equivalent. Send to Intervals as `units: "hr" / "watts" /
"sec_per_km"` (or whatever Intervals supports).

### 3.2. Intervals.icu units the API accepts

Verified via Intervals.icu workout_doc schema (see `intervals_icu_openapi.json`):

| Target | %-units | Absolute units |
|---|---|---|
| HR | `%lthr`, `%max` | `hr` (bpm) |
| Power | `%ftp`, `%mmp` | `watts` |
| Pace | `%pace` | `sec_per_km` (Run), `sec_per_100m` (Swim) |

Intervals propagates absolute targets verbatim into FIT export — watch
receives exact bpm/watts/sec values, no local-LTHR translation needed.

### 3.3. Semantic preservation (HISTORICAL — superseded by §12 Attempt 3b)

Originally scoped to dual-write zone label + absolute bpm into `step.text` so
Intervals UI keeps coach-readable zones while the watch sees absolute targets.
After Attempt 3b, this trade-off is moot: `{units: "%lthr", start, end}` gives
Intervals the zone semantic AND watches the absolute corridor in one shape.

```python
# Original proposal — no longer used:
step["text"] = "Z2 main (85-89% LTHR → 146-153 bpm)"
step["hr"] = {"units": "hr", "value": 146, "end": 153}
```

Current production shape:

```python
step["text"] = "Z2 main"
step["hr"] = {"units": "%lthr", "start": 85, "end": 89}
```

Intervals UI shows the descriptive text, watch sees absolute target. Атлет
in chat / preview gets both numbers — system continues to render zones
relative to current thresholds. Если intervals.icu UI displays its own
«Zone X» indicator next to the bpm range — bonus, не критично.

### 3.4. Fallback for cold-start athletes

If `athlete_settings.{lthr, ftp, threshold_pace}` is None (new athlete
без synced Intervals settings), converter returns the step unchanged with
original `%X` units. Watch falls back to current behavior (uses its local
config). Logging emits a warning so we can surface «complete Intervals
sync to enable precise zones» in onboarding flow.

## 5. Trade-offs

| Aspect | Status quo (`%X`) | Proposed (absolute) |
|---|---|---|
| Watch-agnostic | ❌ relies on local LTHR | ✅ independent |
| Coach-readable Intervals UI | ✅ «Z2 85% LTHR» | 🟡 «146 bpm» (mitigated by description) |
| Multi-athlete template reuse | ✅ same `%lthr` template per all | ❌ template becomes user-specific |
| LTHR update propagation | Watch picks up next sync | Need workout regen if zone changes mid-cycle |
| Implementation cost | 0 | ~50 LoC + tests |
| Auto-detect / per-profile bugs | Silent fail | ✅ caught at server boundary |

**Hardest trade-off:** LTHR change mid-cycle. If `actor_update_zones`
fires Tue after ramp-test → user's LTHR в `athlete_settings` updates
172 → 175. But Wed-Sun workouts already pushed to Intervals with bpm
computed on LTHR=172. Watch alerts на стales values.

Mitigation options:
1. **Live with it.** Drift detector сам fires `actor_update_zones` rarely
   (only когда R²≥0.85 — high confidence). Mid-cycle drift = ~weekly cron
   at most. Stale corridor by ~3-5 bpm для week's remaining workouts —
   acceptable for first iteration.
2. **Auto-regen.** Hook на `actor_update_zones` → re-push already-scheduled
   workouts in current week с обновлёнными бpm. ~50 LoC. Phase B.
3. **Hybrid push.** Send BOTH `%lthr` and absolute. Intervals UI shows
   absolute (computed на push-time), watch uses absolute. If athlete
   re-imports workout to a new watch later, %lthr available для re-computation.
   But Intervals schema might not allow dual fields в single step — verify.

## 6. Phases

### Phase A — server-side converter (cheap win)

**Scope:**
- New method `WorkoutStepDTO.to_absolute_target(...)` resolves `%X` → absolute
- Hook at push boundary: `actor_push_workout` (or wherever serialization)
- Fallback to `%X` when threshold None (cold-start)
- Dual-write description with zone+absolute for coach-readability
- Unit tests: 5-7 cases (each %X type + cold-start + edge: missing threshold per sport)
- Integration test: push synthetic workout, assert Intervals payload contains absolute bpm

**Acceptance:**
- User's Garmin watch displays exact corridor we send, regardless of local LTHR
- Cold-start athlete still gets functional workout (falls back to %X)
- Coach-readable description preserves zone semantic
- No regression on existing pushed workouts (test: same workout before/after produces same Intervals event excluding target units)

**Effort:** ~50 LoC + ~80 LoC tests + spec update. ~1 day.

### Phase B — auto-regen on zone update (only if drift becomes noisy)

**Trigger:** `actor_update_zones` fires (ramp-test → R²≥0.85 → push HRVT2 to
`athlete_settings.lthr`).

**Action:** identify scheduled workouts с push'нутыми bpm values for current
week (Wed-Sun after Tue ramp-test) → re-compute via new threshold → push update
to Intervals.

**Defer until** we see real complaints в production. Most ramp-tests happen
on weekends (push Sun → applies to next Mon week, no mid-cycle workouts
affected).

**Effort:** ~50 LoC + tests. ~half-day.

### Phase C — semantic dual-write polish

**Scope:** improve description format for coach-readable Intervals UI display.
Variants:

- `"Z2 main"` → `"Z2 main (85-89% LTHR → 146-153 bpm)"` — verbose
- `"Z2 main (146-153 bpm)"` — concise
- Localization (ru / en) per athlete

Bikeshed: choose format after Phase A in production. Маленький UX iteration.

**Effort:** ~20 LoC. Spike + iterate.

## 7. Open questions

1. **Intervals.icu schema for absolute HR corridor — ✅ RESOLVED 2026-05-12, POSITIVE outcome.**
   Initial investigation chased «units name» (`bpm`, `start_bpm`, `BPM`) and
   then misread an Intervals.icu UI dropdown as evidence of «no absolute
   channel exists». Both leads were wrong.

   **Actual answer:** the corridor lower bound is encoded with field name
   **`start`**, not `value`. Schema for a HR corridor (values are %LTHR
   percentages — Intervals computes absolute bpm via athlete's stored LTHR
   at FIT-export time):

   ```json
   {"hr": {"units": "%lthr", "start": <pct_low>, "end": <pct_high>}}
   ```

   E.g. `{start: 75, end: 82}` with athlete LTHR=172 → Intervals exports
   absolute corridor 129-141 bpm in FIT, watch displays exact range.
   When the payload uses `value` (legacy in our codebase), Intervals API
   accepts it (lenient validation) but FIT-export interprets it as Lap-HR
   point target with zone clamping — which is exactly the «drift» symptom
   we diagnosed as watch-side LTHR mismatch.

   Verified end-to-end on user 1 watch face 2026-05-12 — see §12 Attempt 3b.

   The UI dropdown that confused us (`нет` / `%МаксЧСС` / `%LTHR` / `зона`) is
   the **single-value** selector. A separate UI affordance produces ranged
   corridors and that path stores `{start, end}` — visible only via API
   readback, not as a labelled dropdown option.

2. **Pace target conversion** — Run pace step может быть в `min/km`
   (athlete-facing) or `sec/km` (machine-readable). Intervals accepts which?
   `%pace` → absolute requires `run_threshold_pace_sec_per_km` (already in
   `athlete_settings.run.threshold_pace`). Swim similarly via CSS.

3. **Power target for Ride** — `%ftp` → watts is trivial. But Intervals
   might support `%cp` (Critical Power) too — should we support both
   inputs, or normalize to `%ftp` at Claude level? **Recommended:** keep
   `%ftp` only as input (current convention), `%cp` is rarely used in chat
   prompts.

4. **Cold-start UX** — if athlete has no LTHR yet, we ship `%lthr` which
   watch interprets via local config (likely auto-detected, may be wrong).
   Should we **refuse to push** workout in this state? Or warning + push as-is?
   **Recommended:** warning + push. Fresh athletes need workouts; «refuse»
   is worse UX than «slightly wrong zones for first 1-2 weeks until ramp test».

5. **Multi-athlete template reuse** — does anyone share workout templates
   across athletes? Currently nobody (per-athlete `compose_workout`), so
   absolute targets don't break anything. Future coaching feature might want
   templates — leave that bridge until we cross it. Можно сохранять `%X`
   в нашем DTO даже после absolute push (so audit / re-export possible).

## 12. Empirical findings 2026-05-12 — investigation log

Four schema candidates tested on user 1 over 2026-05-12. First three failed
in distinct ways; fourth (`{start, end}` corridor) succeeded and is the
shipped fix. Each followed the full verification protocol (push → API
readback → Garmin Connect sync → watch face inspection).

**Test fixture** — 4-step Run, LTHR=172, expected absolute bpm corridors:

| # | Step | %LTHR | Expected bpm |
|---|---|---|---|
| 1 | Warm-up easy | 75-82% | 129-141 |
| 2 | Z2 main aerobic | 85-89% | 146-153 |
| 3 | Z2 upper progression | 89-94% | 153-162 |
| 4 | Cool-down | 70-80% | 120-138 |

### Attempt 1: `{"hr": {"units": "bpm", "value": 129, "end": 141}}`

- ✅ JSON API accepted, stored verbatim (API round-trip confirmed).
- ❌ Garmin watch displayed **214-249 bpm** for warm-up. Physiologically
  impossible (max HR = 191). FIT-export mis-translated `units: "bpm"` —
  values reinterpreted as percent-of-something-else (no clean linear formula
  matched). **Not viable.**

### Attempt 2: `{"hr": {"start_bpm": 129, "end_bpm": 141}}` (no `units` key)

Field names lifted from Intervals.icu openapi schema (which mentioned
`start_bpm`/`end_bpm` in other endpoints — guessed they might apply here).

- ✅ JSON API accepted, stored verbatim (API readback confirmed).
- ❌ Garmin watch displayed **`101`** for all steps. Constant value = watch's
  fallback for «no valid HR target». FIT-export silently dropped the unknown
  schema. **Not viable.**

### Attempt 3a (misleading lead): Intervals.icu UI dropdown survey

Per Attempt 2's residual TODO («create absolute-HR workout via UI builder,
inspect schema»), opened Intervals.icu's web workout builder and inspected
the HR-target dropdown for a Run step. **Available options (full list):**

- `нет` (none)
- `%МаксЧСС` (%MaxHR)
- `%МаксЧСС Диапазон` (%MaxHR range)
- `%LTHR`
- `%LTHR Диапазон` (%LTHR range)
- `зона` (single zone reference)
- `Диапазон зон` (zone range)

**Initial read (wrong):** no absolute-bpm option exists → conclude the feature
is unavailable in the product. We almost shipped a Run-pace-pivot workaround
on this premise.

**Actual fact (per Attempt 3b):** the dropdown is the **units label** picker,
not the value-shape picker. Picking `%LTHR Диапазон` and entering integer
`start`/`end` values produces an absolute-bpm corridor in FIT export — the
`units` label remains `%lthr` cosmetically but the corridor is shipped as
literal bpm. The UI doesn't surface this — only API readback reveals it.

Lesson: UI surveys are useful for elimination but not for confirmation of
absence. The actual schema lived behind the UI in a non-obvious way.

### Attempt 3b: `{"hr": {"units": "%lthr", "start": 75, "end": 82}}` — ✅ **SUCCESS**

After user created «AI: Absolute V2» test workout via Intervals.icu UI builder
(both `value` semantic and ranged corridor), we fetched the saved JSON and
discovered the field name Intervals uses for the corridor lower bound is
**`start`**, not `value`. Our codebase had historically emitted `value` — Intervals
API stored it verbatim (lenient), but FIT export interpreted such payloads as
zone-mapped Lap HR target instead of absolute corridor.

- ✅ JSON API accepted `{start, end}` schema, stored verbatim.
- ✅ Garmin watch (after Garmin Connect sync) displayed **129-141 bpm** for the
  expected 75-82% LTHR corridor at LTHR=172. Match within ±1 bpm rounding.
- ✅ Other steps (146-153, 153-162, 120-138) also matched expected ranges exactly.

**Root cause confirmed:** field naming, not units conversion. Intervals' FIT
export switches behavior based on whether the HR target dict has `value` (point
target → Lap HR mode → watch zone-clamps) vs `start` (corridor → Instant HR
mode → watch displays raw bpm range).

«Мгновенный HR» vs «Круг ЧСС» dropdown in Intervals UI maps to this schema
distinction — visible only at FIT-export layer, not stored as a separate
field. Earlier failed attempts (`units: "bpm"`, `start_bpm`/`end_bpm` direct
fields) were debugging into the wrong layer (units name) when the issue was
field name.

### Code fix applied 2026-05-12

Rename `value` → `start` in three places:

1. `bot/prompts.py` — 5 occurrences in `_zones_block` builder (Run/Ride/Swim
   examples) — Claude now emits `start` directly.
2. `data/intervals/dto.py:_render_target` — read `start` instead of `value` for
   description rendering.
3. `data/intervals/dto.py:PlannedWorkoutDTO._check_steps_have_targets` — accept
   `start` field in validator instead of `value`.

Backwards compat: existing in-DB AI workouts with `value` shape would fail
re-validation if re-pushed; expected — they'd be regenerated by Claude on next
schedule cycle anyway. Already-pushed-to-Intervals workouts on user calendars
not modified.

### Workaround applied earlier 2026-05-12 — bpm corridor in step.text (HISTORICAL, now obsolete)

Between Attempt 2 failure and Attempt 3b success, we ran a stopgap that
embedded absolute bpm directly into the step **text** label:

```python
step["text"] = "Warm-up easy · 129-141 bpm"  # rendered verbatim on watch face
step["hr"] = {"units": "%lthr", "value": 75, "end": 82}  # legacy `value` schema
```

Athlete reads bpm corridor from step label visually; ignores the slightly-off
watch alert range. Reverted after Attempt 3b shipped — the proper `start`/`end`
schema makes both display and alert correct.

**Rollback record:** Each failed attempt caught during pre-workout inspection,
not during actual workout. All reverted within minutes via second
`update_event` call. No broken targets reached an actual training session.
**Risk register update:** any future spike requires verification on watch
face BEFORE the scheduled workout starts (not after).

## 13. Path forward — post-fix follow-ups

§12 Attempt 3b shipped the structural fix (HR corridor via `start`/`end`
schema). Remaining items are optional / nice-to-have rather than blocking.

### 13.1. Run pace convention — OPTIONAL, no longer required

The original pace-pivot proposal (Run aerobic → `% Pace` instead of `% LTHR`)
was scoped as a workaround when we believed absolute HR was structurally
impossible. After Attempt 3b, HR-driven Run is drift-immune and ships exact
bpm corridors to the watch. **The pace-pivot is no longer necessary.**

Still defensible to pursue **as a separate convention choice** (not a drift
mitigation):

- Pace is honest on flat terrain; HR is honest under fatigue/heat. Athletes
  with strong opinions either way may want one as their primary lever.
- Mixed-target sessions (e.g. tempo run with HR ceiling + pace floor) become
  expressible if both are first-class.

Decision deferred. No work scheduled unless an athlete explicitly asks for
the pace-primary convention.

### 13.2. On-device LTHR check — MOOT for HR-driven sessions, still useful as hygiene

The Attempt 3b fix means Intervals computes absolute bpm at FIT-export time
using **its own** stored LTHR (172, kept in sync by `actor_update_zones`).
Watch's local LTHR is no longer in the loop for workout corridors. The
original «check watch-side LTHR after every ramp-test» urgency goes away.

Still worth keeping as one-time hygiene for athletes who:

- Race with HR-zone displays during the event (not following a structured
  workout — uses watch's own zone config).
- Use heart-rate-driven auto-pause / virtual-partner features outside
  Intervals workouts.

Not a blocking issue. Mention once in onboarding; don't pester.

### 13.3. Out-of-scope (not pursuing)

- Power (`%ftp` → watts) and Pace (`%pace` → sec/km, sec/100m) conversion —
  technically feasible but no real-world benefit; both channels already
  drift-immune through their respective stable thresholds, and Attempt 3b
  fix means HR is now in the same drift-immune bucket.
- Multi-vendor (Coros/Wahoo/Polar) FIT interpretation testing — Garmin
  proven sufficient for current athlete base. Re-open if onboarding a
  non-Garmin user.

### 13.5. Decision confirmed 2026-05-12: keep `%X`, не пушить абсолютные

После всех проб подтверждаем — Phase A («server-side `%X → absolute` converter»)
ship'ить **не будем**. `start/end` корридор закрыл root cause без перевода в
bpm/watts/sec. Аргументы за status-quo:

| | `%lthr/%ftp/%pace` (current) | Absolute (`bpm/watts/sec`) |
|---|---|---|
| Watch alerts корректные | ✅ через `start/end` corridor | ✅ |
| LTHR-update propagation | ✅ auto через FIT sync (Intervals считает на лету) | ❌ нужен backfill regen scheduled workouts |
| Coach-readable Intervals UI | ✅ `85-89% LTHR` | 🟡 `146-153 bpm` без зональной семантики |
| Coupling | ✅ renderer pure, threshold не пробрасывается через callers | ❌ ~50-100 LoC threading из `AthleteSettings` |
| Cold-start athletes | ✅ работает без synced settings | ❌ нужен fallback path |

### 13.6. Открытые quirks Intervals UI, найденные 2026-05-12

Не блокирующие, не наш баг — задокументированы чтобы не возвращаться к
ним в попытке «починить» через API.

**A. Swim planned-workout pace display всегда в `min/km`.** Хардкод в Intervals
UI для swim event detail view. Игнорирует `sport_settings.pace_units:
"SECS_100M"` (которое применяется к activity-зонам и статам, но не к workout
view). Подтверждено сравнением вручную-созданной (`109805618`, store
`{units: "secs/100m", value: 142}`) и нашей AI-pushed (`109762368`,
`{units: "%pace", start: 65, end: 78}`) — обе рендерятся как `XX:XX/km` в
UI workout detail. Storage у обеих корректен, разница только на render
layer. Fix невозможен с нашей стороны — нужен report в Intervals support.

**B. `update_event` НЕ триггерит async-enrichment.** Verified empirically: тот
же payload (с одинаковыми `start/end` `%pace` корридорами) через
`POST /events` обогащается полностью (`workout_doc` содержит
`zoneTimes/normalized_power/polarization_index/strain_score/variability_index/locales`),
через `PUT /events/{id}` — только `steps` без enrichment-полей.
`moving_time` и top-level `distance` на event объекте считаются в обоих
случаях, но per-zone breakdown появляется только на create. Implication
для backfill: для cosmetic-rename'ов вроде нашего `value → start` это
неважно (storage корректен, watch получает FIT с правильным корридором),
но если нужен enrichment для UI/dashboard аналитики — потребуется
delete+create вместо update.

**C. ⏳ Bike power discrepancy — pending investigation.** Reported 2026-05-12,
требует повторения той же end-to-end-проверки (push → API readback → Garmin
Connect sync → watch face) что для HR в §12. Параллели с HR-кейсом возможны
(`{units: "%ftp", start: X, end: Y}` уже использует правильный corridor
schema после rename), но нужна верификация на велосипеде с реальным
power meter / smart trainer ERG. Триггер для investigation — следующая
запланированная AI bike-тренировка (когда атлет пушнёт и поедет).

### Анализ задачи реализации фичи из [Issue](https://github.com/radikkhaziev/triathlon-agent/issues/31)

#### Текущий контекст:
В настоящее время метрики CTL (Chronic Training Load) и ATL (Acute Training Load) подтягиваются из сервиса Intervals.icu, где используются временные константы:
- CTL: 42 дня
- ATL: 7 дней

Однако, спортсмены, работающие с TrainingPeaks, сталкиваются с трудностями при сравнении результатов, так как в TrainingPeaks используются:
1. Разные временные константы.
2. Расчёт нагрузок на основе отдельных спортивных метрик TSS (Training Stress Score): rTSS (бег), hrTSS (сердечный ритм), sTSS (плавание), BikeScore/TSS (для велосипеда).

#### Предлагаемая реализация:
1. **Создание расчётного модуля для CTL/ATL/TSB**:
   - Задействовать временные константы τ_CTL (42 дня) и τ_ATL (7 дней) аналогично Intervals.icu.
   - Использовать формулу TrainingPeaks для вычисления нагрузки:
     - rTSS на основе темпа или Lactate Threshold Heart Rate (LTHR).
     - hrTSS с учётом сердечного ритма.
     - sTSS для плавания.
     - TSS для велосипеда.
2. **Унификация данных**:
   - Сохранять посчитанные значения с учетом существующих Intervals.icu значений либо предоставлять доп. метод для расчёта.
3. **Добавление поддержки через инструменты**:
   - Предоставить доступ через функцию `get_training_load` с параметрами источника данных (Intervals.icu или TrainingPeaks).
4. **Интеграция в ответы бота/агента тренера**:
   - Расширить функционал так, чтобы при предоставлении данных о тренировочной нагрузке предлагать оба источника: Intervals.icu и TrainingPeaks.
5. **Документация**:
   - Задокументировать, какая формула применяется для каждого спорта (бег, вело, плавание и т.д.).

#### Хранение данных:
1. **Формат данных**:
   - Хранить TSS конкретного вида (например, rTSS, sTSS) на уровне индивидуальных активностей.
   - Дополнительно сохранять агрегационные данные о CTL/ATL/TSB в разрезе пользователя.
2. **Параметры нормализации**:
   - Убедиться, что для каждого вида спорта корректно применяются нормализационные правила TrainingPeaks.
3. **Структура баз данных**:
   - Учитывать расширяемость для добавления источников данных и метрик.
   - Параметризировать формулы, чтобы поддерживать аккуратность вычислений.
   - Обеспечить хранение промежуточных данных для аудита результатов (особенно для случаев сравнения с UI TrainingPeaks).

### Шаги реализации:
1. **Исследовать формулы TrainingPeaks для TSS по видам спорта**.
2. **Создать сервисную компоненту для расчётов и API для вызова новых метрик**.
3. **Добавить слой агрегированных метрик CTL/ATL/TSB**.
4. **Тестировать, сравнивая результаты с TrainingPeaks UI, для обеспечения точности~5%.**
5. **Доработать интерфейс пользователя и обучение бота**.

Данная задача также требует начальной спецификации и тестирования формул — этот шаг сложен из-за зависимости от нормализационных данных в TrainingPeaks. Вы готовы приступить к реализации или нужно дополнительное уточнение?
